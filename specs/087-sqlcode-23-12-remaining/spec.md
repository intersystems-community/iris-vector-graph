# Spec 087: SQLCODE -23/-12 remaining cases

## Status: Pending

## Problem

5 crashes: 3× SQLCODE -23 "Label not listed", 2× SQLCODE -12 "A term expected".

### -23 Pattern (crashes 4, 13, 15)

All three share: a node variable (e.g. `n1`) appears in **3 or more** comma-separated
patterns. In the third or later occurrence, the node's `rdf_props` alias (`pN`) or an
edge alias ends up referenced inside a JSON_TABLE argument or subquery where it's
not in scope.

Example (crash 15, simplified):
```cypher
MATCH (n0 :L0)-[r0 :T9]->(n1 :L1)-[r1 :T4]->(n2 :L0)-[r2 :T13]-(n3 :L1),
      (n1 :L1{k6:205})-[r3 :T2]->(n4),
      (n5 :L3{k21:false})-[r4 :T3]->(n1)-[r5 :T12]-(n6 :L3),
      (n3{k7:false})<-[r6 :T2]-(n7 :L1)
RETURN n0.id
```
Node `n1` appears in patterns 1, 2, and 3. Node `n3` appears in patterns 1 and 4.
The inline property `{k7:false}` on `n3` in pattern 4 requires a `rdf_props` JOIN —
if this JOIN's `?` param ends up positioned incorrectly relative to a JSON_TABLE
argument in the generated SQL, IRIS reports -23.

### -12 Pattern (crashes 10, 14)

Both involve:
- Multi-comma MATCH (10+ paths)
- A `RETURN` with mixed aggregates (`sum()`, `count()`) and edge properties (`r0.k117`)
- A complex boolean WHERE with `{k22:false}` inline node properties

The generated Stage1 CTE may not expose the right columns for the outer SELECT,
producing `SELECT FROM (Stage1)` with an empty select list.

Example (crash 10 simplified):
```cypher
MATCH (n0 :L1 :L0)-[r0 :T7]-(n1)...(complex 10-path)...
WHERE (r0.k124 = 'test') AND ...
RETURN (r0.k117) AS a0, sum(1614595245) AS a1, count(-1) AS a3 ORDER BY a0
```

## Root Cause Hypotheses

### For -23:
The THIRD occurrence of a node that already has a `rdf_props` JOIN from pattern 1
generates ANOTHER `rdf_props` JOIN (`p_new`) for its inline props in pattern 3.
But `p_new` gets positioned incorrectly relative to a CROSS JOIN or JSON_TABLE
that was inserted for an UNWIND somewhere in the 10+ pattern chain.

### For -12:
When an edge variable `r0` is accessed as `r0.k117` in RETURN after a Stage1
generated by a `WITH` clause, and `r0` was from an undirected edge in the MATCH,
the Stage1 CTE stores it as `e._p AS r0` but the RETURN's `translate_return_clause`
generates `Stage1.r0` which doesn't exist as a column (Stage1 has `r0` from `e._p`,
but some code path emits an empty select when `r0.k117` requires property lookup
on the stage alias).

## Implementation Notes

### -23 fix:
In `translate_relationship_pattern`, when processing a node that's ALREADY BOUND
(second/third occurrence), its inline property JOINs should be appended to
`join_clauses` at the CORRECT position — specifically, they must come BEFORE
any subsequent CROSS JOIN for UNWIND. Currently the property JOINs may be
appended after an earlier CROSS JOIN, putting them out of IRIS's scope.

**Fix approach**: maintain a "last regular join position" index and insert
inline-property JOINs for already-bound nodes at that position, not at the end.

### -12 fix:
In `translate_return_clause`, when the RETURN item is `PropertyReference(r0, k117)`
and `r0`'s alias is a Stage alias (`Stage1`), the property lookup should use
`Stage1.r0_k117` IF that column was projected into Stage1, OR fall back to a
direct `SQLUser.JSON_VALUE(StageN.r0, '$.k117')` — but only if Stage1 exposes
the raw edge alias value.

Actually the simpler fix: when `WITH n2, r2, ...` puts `r2` into Stage1 as column
`r2`, and RETURN accesses `r2.k117`, translate it as `SQLUser.JSON_VALUE(Stage1.r2, '$.k117')`.
